# Company OG Image Cache

Company Open Graph images are cached outside Vercel build artifacts in
Cloudflare R2. The goal is to avoid regenerating hundreds of mostly stable
PNG cards during every `next build`, while still rerendering them when the
renderer code changes.

## Cache Key

Objects are written to:

```text
og/company/<renderer-version>/<locale>/<slug>.png
```

`<renderer-version>` is injected by `next.config.ts` as
`COMPANY_OG_RENDERER_VERSION`. It is a SHA-256 hash, truncated to 16 hex
characters, of a conservative input set:

- `app/[lang]/(app)/company/[slug]`
- company data/action helpers that feed the route
- `src/lib/og`
- `public/fonts/JetBrainsMono-Bold.ttf`
- `package.json`, `pnpm-lock.yaml`, and `next.config.ts`

The input set intentionally over-invalidates. A nearby company route change
can create a new renderer version even if pixels do not change. That is safer
than under-invalidating and serving stale social cards after a real renderer
change.

## Runtime Flow

For `/[lang]/company/[slug]/opengraph-image-*`:

1. Compute `og/company/<renderer-version>/<locale>/<slug>.png`.
2. If R2 is configured and `COMPANY_OG_CACHE_BYPASS` is not enabled, read R2.
3. On hit, return the PNG immediately. This avoids Typesense/Postgres reads.
4. On miss, load the company, render `ImageResponse`, upload the PNG to R2,
   and return the generated bytes.
5. Unknown company slugs render the not-found card and are not written to R2.

If R2 env vars are absent, the route behaves like the old direct renderer.
To get cross-build reuse in production, Vercel must expose the existing R2
environment variables to both build and runtime:

- `R2_ENDPOINT_URL`
- `R2_ACCESS_KEY_ID`
- `R2_SECRET_ACCESS_KEY`
- `R2_BUCKET`

### Turborepo Env Allowlist

Vercel project env vars are not automatically visible inside `pnpm turbo run
build`. Turborepo filters the build environment to the variables listed in the
root `turbo.json` task env allowlist. This has caused recurring production
deploy failures: the R2 vars existed in Vercel, but Turbo stripped them from
`@jobseek/web#build`, so the OG route could not read R2, regenerated cold
company cards, hit Typesense 429s, and failed prerendering.

When adding or renaming any env var read during `next build`, update
`turbo.json` in the same PR. For this cache, the build allowlist must include:

- `R2_ENDPOINT_URL`
- `R2_ACCESS_KEY_ID`
- `R2_SECRET_ACCESS_KEY`
- `R2_BUCKET`
- `COMPANY_OG_CACHE_BYPASS`
- `COMPANY_OG_PRERENDER_TOP_N`
- `COMPANY_OG_RENDERER_VERSION_SALT`

Treat this Vercel/Turbo warning as a release blocker:

```text
environment variables are set on your Vercel project, but missing from "turbo.json"
```

## Force Controls

Use `COMPANY_OG_PRERENDER_TOP_N` to adjust how many high-traffic company
cards are rendered during `next build`. It defaults to `200`; set it to `0`
only for build-classification CI lanes that must not depend on external
company data:

```bash
COMPANY_OG_PRERENDER_TOP_N=0 pnpm --filter @jobseek/web build
```

Use `COMPANY_OG_RENDERER_VERSION_SALT` to force a new namespace at build time:

```bash
COMPANY_OG_RENDERER_VERSION_SALT=rerender-2026-06-16 pnpm --filter @jobseek/web build
```

This changes `<renderer-version>`, so every requested/prebaked company card
misses R2 and writes a new object. Use this for intentional global rerenders.

Use `COMPANY_OG_CACHE_BYPASS=1` to skip R2 reads and overwrite the current key:

```bash
COMPANY_OG_CACHE_BYPASS=1 pnpm --filter @jobseek/web build
```

This is useful when a bad object exists under the current renderer version.
Because the public response is CDN-cached, pair runtime bypass with a deploy
or CDN purge if already-served route responses need immediate replacement.

Both controls can be combined:

```bash
COMPANY_OG_RENDERER_VERSION_SALT=rerender-2026-06-16 \
COMPANY_OG_CACHE_BYPASS=1 \
pnpm --filter @jobseek/web build
```

## Build Behavior

`generateStaticParams` still fetches the top company slugs from Typesense so
popular cards are prebaked. During the actual OG render step, cache hits come
from R2 and do not call `getCompanyBySlug`. That keeps routine builds from
issuing hundreds of company detail reads and prevents Typesense 429s from
cascading into Postgres fallback load.

If Typesense rate-limits the top-slug query, the query retries in place. If it
still fails in production with Typesense configured, the build fails loud
instead of silently shipping a zero-prebake deploy.

## Retention

Renderer-versioned keys intentionally leave old namespaces behind. That makes
rollbacks and CDN revalidation safe, but it also means the R2 bucket would grow
forever without cleanup.

The repo provides `apps/web/script/prune-company-og-cache.ts`, exposed as:

```bash
pnpm --filter @jobseek/web og:prune -- --retain-versions 8 --min-age-days 60
```

The script is dry-run by default. It groups objects under `og/company/` by
renderer version, keeps the newest N namespaces, and only deletes older
namespaces whose newest object is past the minimum age. Pass `--yes` to delete:

```bash
pnpm --filter @jobseek/web og:prune -- \
  --yes \
  --retain-versions 8 \
  --min-age-days 60 \
  --max-delete 20000
```

`.github/workflows/prune-company-og-cache.yml` runs this weekly against the
production R2 bucket with those defaults. The `--max-delete` cap is intentional:
if object volume unexpectedly spikes, the workflow fails loudly instead of
deleting an unbounded number of objects.

## Tradeoff

The cache key is renderer-versioned, not company-data-versioned. Company name,
logo, description, and count changes can leave an existing OG image stale until
the renderer version changes or a force control is used. That is an intentional
cost of avoiding churn from frequently changing company data.
