# i18n Guidelines

## Explicit IDs (mandatory)

Every translation must use a manually assigned, semantically rich `id`. Never rely on Lingui's auto-generated hashes.

IDs use **dot-namespaced** format: `{page}.{section}.{element}`

```
home.hero.title
home.hero.subtitle
dashboard.settings.notificationsLabel
pricing.pro.cta
common.nav.signIn
```

## Comments (mandatory)

Every translation must include a `comment` that gives translators enough context to produce an accurate translation. Describe where the string appears, what it refers to, or any constraints (e.g. character limits).

## Syntax

### JSX — `<Trans>`

```tsx
<Trans id="home.hero.title" comment="Main heading on the landing page">
  Find your next opportunity
</Trans>
```

### JavaScript — `t()`

```ts
t({
  id: "home.hero.title",
  comment: "Main heading on the landing page",
  message: "Find your next opportunity",
})
```

### Plurals — `plural()`

```ts
plural(count, {
  id: "dashboard.feed.jobCount",
  comment: "Number of matching jobs shown above the feed",
  one: "# job found",
  other: "# jobs found",
})
```

## Naming conventions

| Segment   | Rule                                  | Examples                          |
| --------- | ------------------------------------- | --------------------------------- |
| page      | Route or feature name                 | `home`, `dashboard`, `pricing`    |
| section   | Logical area within the page          | `hero`, `features`, `settings`    |
| element   | Specific piece of text                | `title`, `cta`, `description`     |
| `common.` | Prefix for strings reused across pages | `common.nav.signIn`, `common.footer.copyright` |

Keep IDs **lowercase**, use **camelCase** only for the element segment when needed (`notificationsLabel`).

## What not to do

```tsx
// Missing id — will get an auto-generated hash
<Trans>Find your next opportunity</Trans>

// Missing comment — translators lack context
<Trans id="home.hero.title">Find your next opportunity</Trans>

// Vague id — not semantically useful
<Trans id="text1" comment="...">Find your next opportunity</Trans>
```

## Pre-commit gate: i18n coverage

`scripts/check-i18n-coverage.sh` runs on every commit that touches
`apps/web/{app,src,locales}/` (configured in `.pre-commit-config.yaml`).
It enforces two invariants:

1. **No empty `msgstr` in de/fr/it.** Adding a `<Trans>` / `t({…})` /
   `plural({…})` macro extracts the source-locale message into `en.po`
   and leaves an empty placeholder in the non-source catalogs. Failing
   to translate it shows the English fallback to non-English users in
   production. The check greps for `msgstr ""` (excluding the file
   header) and fails if any non-source catalog has one.

2. **Catalogs in sync with source code.** Even with #1 green, a fresh
   `<Trans>` that hasn't been put through `pnpm extract` yet would
   pass — the catalogs simply don't know about it. The hook runs
   `pnpm --dir apps/web extract` in a sandboxed snapshot of the
   catalogs and compares the *sorted set of msgids* against the
   committed catalogs. Any msgid only in source (newly added macro
   that wasn't extracted) or only in the catalog (orphan macro that
   should have been pruned by extract) fails the commit. The
   positional content of the .po file is *not* compared — extract's
   ordering is filesystem-dependent and shuffles harmlessly across
   runs.

When the gate fires:

- **Empty msgstr** — open `apps/web/locales/{de,fr,it}.po`, find the
  listed key, fill in `msgstr "..."`, re-stage, re-commit.
- **Stale catalog** — run `pnpm --dir apps/web extract` to refresh the
  .po files, then translate any newly-extracted entries and re-commit.

Escape hatches (last resort):
- `git commit --no-verify` — bypasses every pre-commit hook. Don't use
  casually; untranslated strings should never reach `main`.
- `SKIP_STALE_CHECK=1 git commit ...` — skips only the stale-catalog
  check while keeping the empty-msgstr check active. Useful when the
  Lingui extractor itself is broken in a way unrelated to your change
  (rare).

The stale-catalog check is skipped automatically when `pnpm` or the
local `node_modules` directories are missing — the empty-msgstr check
remains the gatekeeper in those environments.

## Adding a new language

1. **`src/lib/i18n.ts`** — add the locale code to the `locales` array:
   ```ts
   export const locales = ["en", "de", "fr", "it", "es"] as const;
   ```

2. **`lingui.config.ts`** — add the same code to the `locales` array so the CLI knows to extract/compile it.

3. **`app/layout.tsx`** — update the inline `langScript` to include the new code in its allow-list:
   ```ts
   const langScript = `try{var l=location.pathname.split('/')[1];if(['en','de','fr','it','es'].includes(l))document.documentElement.lang=l}catch(e){}`;
   ```

4. **`proxy.ts`** — add the new code to the matcher so locale-prefixed paths skip the proxy:
   ```ts
   matcher: ["/((?!_next|api|flags|fonts|publicdomain|favicon\\.ico|en|de|fr|it|es|.*\\..*).*)" ],
   ```

5. **Create the catalog** — run `pnpm extract` from `apps/web`. This creates `locales/es.po`. Translate the strings.

6. **Compile & verify** — run `pnpm build` and check the build output confirms the new locale pages are generated.
